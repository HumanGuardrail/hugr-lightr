# Feature parity — everything Docker does, weighing nothing

- **Status:** Owner directive #2, 2026-06-11. Companion to
  `performance-bar.md` (the two axes of the product: full parity × zero
  weight).
- **Owner's words (verbatim, pt-BR):** "é muito importante que entregue
  todas as features e funcionalidades do docker, e mais ainda se possível,
  e que seja insano de light, warp speed absoluta, tem que rodar até em um
  micro-ondas."

## The reconciling law (architecture-level, inviolable)

> **Every feature exists; no feature weighs anything until invoked.**
> Idle cost of any feature = 0. Features are materialized on demand,
> exactly like data. A feature that requires a resident process to exist
> is redesigned until it doesn't — or it integrates the OS's own resident
> machinery instead of shipping ours.

Docker's weight is not any single feature — it is that **all features are
resident all the time** (dockerd + VM + network/storage drivers, awake
24/7). Parity without weight means parity in *capability*, never in
*residency*.

## The one physics note (stated honestly)

A Linux container needs a Linux kernel. On macOS there are exactly two
ways: a VM (Docker keeps one warm 24/7 — that's the 4 GB) or no Linux.
Lightr's answer: an **ephemeral microVM** (Virtualization.framework) that
boots in well under a second for the run that needs Linux **and dies with
it**. Parity: yes, you can run any Linux image. Weight: only while it runs,
~0 idle. Physics is paid per use, never per install.

## The rings (capability grows in rings; the core never gains weight)

### R0 — the warp core (v0.1, current scope)
| Docker | Lightr | Mechanism |
|---|---|---|
| (no equivalent — Docker re-runs everything) | memoized `run` | Action Cache replay, ≤10 ms |
| `docker commit` | `lightr snapshot` | stat-index + BLAKE3, ≤100 ms warm |
| `docker pull` (local hit) | `lightr hydrate` | CoW clonefile, ≤100 ms/GB |
| — | `lightr status` | index diff |

### R1 — runtime parity (native tier)
| Docker | Lightr | Weight note |
|---|---|---|
| `run -v` volumes/binds | dirs + CoW clones; named volumes = store refs | zero (it's the store) |
| `run -e/-w/-u`, workdir, env | process spawn args | zero |
| `exec` / `logs` / `ps` / `stop` | per-run control files under `~/.lightr/run/<id>/`, owned by the run's process tree, gone when it exits | **no daemon**: state lives with the process, not above it |
| resource limits | rlimits/taskpolicy (native), VM config (vz) | zero idle |
| rootless | native is rootless by construction | — |

### R2 — the Linux tier (parity with Docker's reason-to-exist on a Mac)
| Docker | Lightr | Weight note |
|---|---|---|
| run Linux images on macOS | `vz` ephemeral microVM per run (or per explicit session) | boots <1 s, dies with the job; 0 idle |
| `pull`/`push` OCI registries | `lightr oci import/export` — image ↔ store conversion (layers unpacked once into file-level CAS) | conversion paid once; after that it's CoW objects |
| `-p` port publish, container DNS/networks | vmnet/virtio NAT configured on the ephemeral VM at spawn | exists only while the VM exists |

### R3 — ecosystem parity (meet devs where they are)
| Docker | Lightr | Weight note |
|---|---|---|
| `docker build` (Dockerfile) | `lightr build`: Dockerfile-compat parser; **each step = a memoized run keyed by content** | see "beyond" — this is where we don't match, we obliterate |
| `docker compose` | `lightr compose up`: same binary parses compose.yml, schedules the DAG of runs | a parser, not a service; zero idle |
| restart policies / `--restart` | integrate the OS supervisor (launchd/systemd units generated by `lightr`) | **the OS already has a daemon; we don't ship a second one** |
| healthchecks, secrets, configs | run-spec features (files/env, store-backed) | zero idle |
| Docker socket API (testcontainers et al.) | opt-in `lightr compat-socket` — an **ephemeral** API shim that lives only for the duration of the invoking session | the only "daemon" in the product: opt-in, scoped, self-terminating |

### R4 — "e mais ainda" (the ring Docker cannot enter)
- **Memoized builds, cross-machine** — every build step is a content-keyed
  run: incremental builds are perfect by construction, and with Stage 2 the
  teammate's build IS your cache. BuildKit's fragile local cache vs a
  content-addressed action cache: not a fair fight.
- **Instant workspace time-travel** — refs have lineage (`parent` in the
  RefRecord); `lightr hydrate @app/web@yesterday` is the same ≤100 ms CoW op.
- **Run-state snapshot/restore** (vz tier) — resume a booted environment in
  ~ms; `docker start` boots, Lightr *resumes*.
- **Scale-to-zero cloud** — the same refs execute on Runners (fc microVMs);
  local and cloud are one store, one law.

## Non-goals that survive both directives
- **Swarm/orchestration** — dead weight even inside Docker; cluster
  scheduling belongs to Runners' fabric. Refused.
- **A resident daemon of our own** — under any feature pressure, forever.
  (The compat-socket shim is ephemeral and opt-in; launchd/systemd
  integration delegates residency to the OS.)

## The microwave clause (minimum-hardware floor, testable)

Lightr must run, full R0/R1 capability, on:
- **1 CPU core, 512 MB RAM, any POSIX filesystem** (CoW where available —
  APFS/btrfs/XFS; graceful byte-copy fallback elsewhere, slower but
  correct, and the bench harness reports which mode you got);
- **static binary ≤ 10 MB**, macOS arm64 + Linux x86_64/arm64 (musl);
- **zero background processes** at all times (`ps` proves it — already
  acceptance item A4).

If it has a POSIX kernel and half a gig of RAM, it runs. A Raspberry Pi
Zero 2 is an honest CI runner; a microwave with Linux is, regrettably for
the microwave, in scope.
